# CONTEXT — 智能发票识别与自动报销填单系统

## 领域术语

- **增值税发票**：系统唯一处理的发票类型。包含电子发票和纸质增值税专用发票/普通发票，具有标准化版面（左上二维码 + 表格结构）和固定字段集合（发票号码、日期、购买方、销售方、金额、税额、价税合计）。出租车票、定额发票、通用机打发票等不在 scope 内。

- **可校验字段**：二维码和 OCR 结果共有的字段——发票号码、日期、不含税金额。双信息校验时逐一比对，全部通过才算校验通过。

- **仅 OCR 字段**：二维码中不包含、只能从 OCR+LLM 提取的字段——购买方、销售方、税额、价税合计。不参与校验，直接信任 OCR+LLM 结果。

- **双信息校验**：逐字段比对二维码解析结果与 OCR+LLM 提取结果中所有可校验字段的过程。全部一致则显示"校验通过"；任一字段不一致则阻塞提交，并在不一致字段旁标红显示两方数据的差异，用户需重新上传或检查原图后重试。

- **OCR 输入**：PaddleOCR 始终对整张原图进行文字识别。二维码的文字量极小，不干扰表格区域的结构化提取。YOLO 裁剪出的二维码仅用于解码，不参与 OCR 流程。

- **LLM 信息抽取**：多模态模式。将原图与 PaddleOCR 输出的文字一同发送给在线多模态大模型，模型同时利用视觉布局信息和 OCR 文本归属字段，输出结构化的键值对（发票号码、日期、购买方、销售方、金额、税额、价税合计）。API 调用层抽象为统一接口，支持通过配置文件在多个后端（GPT-4o、Claude、通义千问、DeepSeek 等）间切换，每个后端封装为自己的适配器。

- **报销表单**：HTML 表单，字段分两类。（1）**自动提取字段**：发票号码、开票日期、购买方、销售方、金额、税额、价税合计——由系统填入，只读。（2）**手动填写字段**：报销人、报销事由、所属部门——用户手填，可编辑。提交日期由系统自动设为当天。

- **发票代码**：增值税发票的编码前缀（12位），与发票号码组合构成全系统唯一标识。重复报销检测以此组合为键。

- **重复报销**：新发票的「发票代码 + 发票号码」组合在已报销记录中已存在时触发。系统报警并阻塞提交。判定仅基于这两个字段。

- **YOLO 模型**：基于 Ultralytics YOLO（默认使用 YOLOv8）预训练权重微调。用于二维码定位与裁剪，使用爬虫采集的增值税发票图片构建数据集，采用 AI 辅助打标 + 人工审核的方式完成标注。

- **Web 框架**：FastAPI + Jinja2 模板渲染。FastAPI 提供异步 HTTP 处理和自动 API 文档，Jinja2 渲染多页面流程（上传 → 识别结果 → 报销表单 → 提交成功/报警）。

- **数据库**：SQLite。单文件零配置，与 FastAPI 随起随用，适配单用户 demo 场景，无需额外数据库服务依赖。

- **报销单导出**：同时支持 Excel (.xlsx) 和 PDF 两种导出格式。提交成功后 HTML 页面上提供两个下载按钮。Excel 用 openpyxl 生成，PDF 用 WeasyPrint 或 reportlab 从 HTML 渲染。

- **上传格式**：接受 JPG、PNG、PDF 三种格式，单文件上限 10MB。PDF 文件通过 PyMuPDF (fitz) 提取首页并转为 PNG 后进入 YOLO+OCR 流水线。

- **训练数据集**：公开数据集（ICDAR、天池等已有标注的增值税发票数据）为主体，百度图片爬取约 100 张作为多样性补充。总计 300-500 张，AI 辅助打标 + 人工审核。

- **数据集分割**：标注完成的图片按 80% 训练集 / 20% 验证集随机分割（固定随机种子 42）。分割由 `train_yolo.py --split 0.8` 自动执行，输出到 `data/dataset/train/` 和 `data/dataset/val/`，每个子目录包含 `images/` 和 `labels/`。

- **页面流程**：5 个独立页面。① 上传页（选择文件 → 上传）→ ② 处理中页（富展示：YOLO 二维码检测框截图 + OCR 原始文字流 + LLM 返回的原始 JSON，各阶段结果随处理进度逐步展示，通过 SSE 推送或轮询实现）→ 校验分流 → ③a 校验失败页（不一致字段标红、阻塞、引导重新上传）或 ③b 报销表单页（7 个只读字段 + 3 个手填字段 + 提交按钮）→ 重复检查分流 → ④a 重复报警页（已报销编号展示、阻塞）或 ④b 提交成功页（Excel/PDF 下载按钮）。

- **二维码缺检降级**：YOLO 未检测到二维码时，系统不阻塞流程。跳过二维码解析和双信息校验，OCR+LLM 提取结果直接进入表单，但页面顶部显示黄色警告"二维码缺失，数据未经校验，请人工核对"。

- **LLM 降级**：LLM API 调用失败时，自动重试 1 次；仍失败则降级为正则规则提取（从 OCR 原始文本中按字段名模式匹配），页面上标注"大模型不可用，已使用规则提取"。

- **默认 LLM 后端**：通义千问 Qwen-VL（阿里云 DashScope API）。配置文件支持切换到 GPT-4o、Claude、DeepSeek 等后端。

- **上传文件存储**：本地 `uploads/` 目录，文件以 `{uuid}.{ext}` 格式命名。图片不进数据库 blob。
